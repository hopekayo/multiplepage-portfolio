🚀 快速入门

安装确认

# 检查是否已安装
claude --version

# 查看帮助
claude --help

启动方式

# 方式 1：在项目目录中直接运行
cd /path/to/your/project
claude

# 方式 2：指定目录启动
claude /path/to/project

# 方式 3：使用别名（推荐）
# 在 ~/.zshrc 中添加：
alias cc='claude'

首次使用

# 1. 登录（如未登录）
claude login

# 2. 开始对话
claude

# 3. 输入你的第一个任务
"帮我分析这个项目的结构"

✍️ 提示词最佳实践

1. 明确具体

# ✅ 好的提示词
"将这个函数的时间复杂度从 O(n²) 优化到 O(n log n)，
使用快速排序算法。保持现有的 API 接口不变。"

# ❌ 差的提示词
"让这个函数更快"

2. 提供上下文

# ✅ 好的提示词
"这是一个 Next.js 14 项目，使用 App Router。
我需要在这个页面中添加服务端渲染：
@src/app/products/page.tsx
考虑我们已经有这些 API：@src/lib/api.ts"

# ❌ 差的提示词
"帮我加个 SSR"

3. 指定输出格式

# ✅ 好的提示词
"生成一个检查清单，包括：
- [ ] 代码质量检查项
- [ ] 测试覆盖检查项
- [ ] 性能检查项
- [ ] 安全检查项

每个检查项后面附上简短说明。"

# ❌ 差的提示词
"给我一些检查建议"

4. 设定约束条件

# ✅ 好的提示词
"重构这个模块，要求：
- 不改变外部 API
- 保持向后兼容
- 代码行数减少 20% 以上
- 添加完整的类型定义"

# ❌ 差的提示词
"改进这个代码"

5. 使用示例

# ✅ 好的提示词
"像这样的格式生成文档：

## 函数名
**功能：** 简短描述
**参数：** 
- param1: 类型 - 说明
**返回：** 类型 - 说明
**示例：**
\`\`\`typescript
// 代码示例
\`\`\`

请为 @src/utils/ 下所有函数生成这样的文档"

⚠️ 常见陷阱与避免方法

1. 上下文过长

问题： 对话历史太长，导致响应变慢或超出限制

解决方案：

# 定期清理
"/clear"

# 开始新任务时明确说明
"开始一个新任务，忽略之前的对话"

# 分多次对话
"我们分几次完成这个功能，今天先做第一部分"

2. 任务过于宽泛

问题： Claude 给的回答太泛，不实用

解决方案：

# 缩小范围
"只关注认证模块，不要涉及其他功能"

# 指定深度
"详细解释这个函数的每一行代码"

# 要求具体示例
"给我 3 个实际的代码示例"

3. 盲目信任输出

问题： Claude 可能生成错误或过时的代码

解决方案：

# 始终验证
"这个方案有什么潜在问题？"

# 要求解释
"为什么选择这个方案而不是其他方案？"

# 交叉验证
"这个 API 在 Next.js 14 中是最新的吗？"

# 测试驱动
"先写测试，再实现功能"

4. 缺乏迭代

问题： 期望一次就得到完美答案

解决方案：

# 第一轮
"给出一个初步方案"

# 第二轮
"这个方案有哪些可以改进的地方？"

# 第三轮
"现在实现改进后的版本"

5. 忽视安全

问题： 生成的代码可能有安全隐患

解决方案：

# 明确要求
"确保代码符合 OWASP 安全最佳实践"

# 安全检查
"审查这段代码的安全漏洞"

# 不暴露敏感信息
# ❌ 不要分享：API Keys、密码、数据库凭证
# ✅ 可以分享：代码结构、错误信息（脱敏后）

💡 个人建议

基于我的使用经验，以下几点最重要：

1. 把 Claude 当同事，不是搜索引擎

  • ✅ 好：「我们一起重构这个模块，你先分析，我review 方案后再实现」
  • ❌ 差：「给我代码」

2. 保持对话的连续性

  • 一个会话专注于一个主题
  • 复杂任务分多个会话完成
  • 定期 /clear 保持上下文清爽

3. 学会「追问」

# 追问原因
「为什么选择这个方案？」

# 追问替代方案
「还有其他实现方式吗？各有什么优缺点？」

# 追问细节
「这个类型定义能再详细点吗？」

4. 建立自己的提示词库

保存常用的提示词模板，例如：

# 代码审查模板
「审查这段代码，关注：
1. 功能正确性
2. 性能问题
3. 代码可读性
4. 类型安全
5. 错误处理

给出具体的改进建议和代码示例」

# 新功能开发模板
「实现 [功能描述]，要求：
- 技术栈：[技术选型]
- 输入：[输入数据]
- 输出：[期望结果]
- 约束：[限制条件]

先设计方案，确认后再实现」

5. 验证、验证、再验证

  • 运行生成的代码
  • 检查测试是否通过
  • 手动验证关键功能
  • 不要盲目信任任何输出
