<h2>⚠️ 常见陷阱与避免方法</h2>
<h3>1. 上下文过长</h3>
<p><strong>问题：</strong> 对话历史太长，导致响应变慢或超出限制</p>

<pre><code class="language-bash"># 定期清理
&quot;/clear&quot;

# 开始新任务时明确说明
&quot;开始一个新任务，忽略之前的对话&quot;

# 分多次对话
&quot;我们分几次完成这个功能，今天先做第一部分&quot;
</code></pre>
<h3>2. 任务过于宽泛</h3>
<p><strong>问题：</strong> Claude 给的回答太泛，不实用</p>
<p><strong>解决方案：</strong></p>
<pre><code class="language-bash"># 缩小范围
&quot;只关注认证模块，不要涉及其他功能&quot;

# 指定深度
&quot;详细解释这个函数的每一行代码&quot;

# 要求具体示例
&quot;给我 3 个实际的代码示例&quot;
</code></pre>
<h3>3. 盲目信任输出</h3>
<p><strong>问题：</strong> Claude 可能生成错误或过时的代码</p>
<p><strong>解决方案：</strong></p>
<pre><code class="language-bash"># 始终验证
&quot;这个方案有什么潜在问题？&quot;

# 要求解释
&quot;为什么选择这个方案而不是其他方案？&quot;

# 交叉验证
&quot;这个 API 在 Next.js 14 中是最新的吗？&quot;

# 测试驱动
&quot;先写测试，再实现功能&quot;
</code></pre>
<h3>4. 缺乏迭代</h3>
<p><strong>问题：</strong> 期望一次就得到完美答案</p>
<p><strong>解决方案：</strong></p>
<pre><code class="language-bash"># 第一轮
&quot;给出一个初步方案&quot;

&quot;这个方案有哪些可以改进的地方？&quot;

# 第三轮
&quot;现在实现改进后的版本&quot;
</code></pre>

<p><strong>解决方案：</strong></p>
<pre><code class="language-bash"># 明确要求

<h2>💡 个人建议</h2>
<p>基于我的使用经验，以下几点最重要：</p>
<h3>1. 把 Claude 当同事，不是搜索引擎</h3>
<ul>
<li>✅ 好：「我们一起重构这个模块，你先分析，我review 方案后再实现」</li>
<li>❌ 差：「给我代码」</li>
</ul>
<h3>2. 保持对话的连续性</h3>
<ul>
<li>一个会话专注于一个主题</li>
<li>复杂任务分多个会话完成</li>
<li>定期 <code>/clear</code> 保持上下文清爽</li>
</ul>
<h3>3. 学会「追问」</h3>
<pre><code class="language-bash"># 追问原因
「为什么选择这个方案？」

# 追问替代方案
「还有其他实现方式吗？各有什么优缺点？」

# 追问细节
「这个类型定义能再详细点吗？」
</code></pre>
<h3>4. 建立自己的提示词库</h3>
<p>保存常用的提示词模板，例如：</p>
<pre><code class="language-bash"># 代码审查模板
「审查这段代码，关注：
1. 功能正确性
2. 性能问题
3. 代码可读性
4. 类型安全
5. 错误处理

给出具体的改进建议和代码示例」

# 新功能开发模板
「实现 [功能描述]，要求：
- 技术栈：[技术选型]
- 输入：[输入数据]
- 输出：[期望结果]
- 约束：[限制条件]

先设计方案，确认后再实现」
</code></pre>
<h3>5. 验证、验证、再验证</h3>
<ul>
<li>运行生成的代码</li>
<li>检查测试是否通过</li>
<li>手动验证关键功能</li>
<li>不要盲目信任任何输出</li>
</ul>
